> 이 문서는 Claude Code가 프로젝트 컨텍스트로 사용하는 스펙 문서입니다.
> 프로젝트 루트에 `CLAUDE.md`로 두고 개발을 진행하세요.
> 레포/코드네임은 siso로 확정. 최종 서비스명·도메인은 미정이므로

## 0. 프로젝트 한 줄 요약

좌파 성향 커뮤니티와 우파 성향 커뮤니티의 글을 한 화면에서 나란히 모아보고,
같은 주제에 대한 양쪽 시각을 비교하며, 익명으로 토론할 수 있는 웹 서비스.
수익 모델은 디스플레이 광고(구글 애드센스).

## 1. 핵심 설계 원칙 (모든 기능 구현 시 최우선 적용)

1. **완전한 대칭성**: UI, 알고리즘, 노출 순서, 광고 배치 등 모든 요소에서
   좌·우가 구조적으로 동등해야 함. 어느 한쪽이 위/먼저/크게 보이는 설계 금지.
2. **법적 리스크 최소화**:
   - 크롤링/수집 시 robots.txt 및 각 사이트 이용약관 준수
   - 원문 전체 복제 금지 — 제목 + 요약(공백 포함 200자 이내) + 원문 링크만 저장/노출
   - 모든 글에 출처(커뮤니티명, 원문 링크) 명시
   - 개인정보처리방침 및 이용약관 페이지 필수
3. **익명 기반, 회원가입 없음**: 가입 절차 없이 누구나 읽고 쓸 수 있음.
   대신 어뷰징 방지 장치(아래 7절)를 촘촘히 설계.
4. **비용 최소화 우선**: 무료 티어로 시작, 트래픽 성장 시 단계적 업그레이드.

## 2. 사용자(방문자)가 할 수 있는 행동 전체 목록

### 2.1 읽기
- 좌 성향 커뮤니티 글 피드 보기 (제목, 요약, 출처, 원문 링크, 수집 시각)
- 우 성향 커뮤니티 글 피드 보기 (동일 구조)
- 플레이그라운드(중립 존)에서 같은 주제로 매칭된 좌/우 글 쌍을 병렬 비교 보기
- 주제별/커뮤니티별/시간별 필터 및 정렬
- 인기 토론(댓글 수, 참여자 수 기준) 목록 보기
- 원문 링크 클릭 → 새 탭으로 원 커뮤니티 이동

### 2.2 참여 (모두 익명)
- 글 쌍 또는 개별 글에 댓글 작성
- 댓글에 대댓글 작성 (2 depth까지만 허용 — 무한 스레드 방지)
- 댓글 추천 / 비추천 (세션당 댓글별 1회)
- 댓글 신고 (사유 선택: 욕설·인신공격 / 혐오 표현 / 도배·스팸 / 기타)
- 글 쌍에 대한 "내 입장" 투표 (좌에 공감 / 우에 공감 / 중립) — 투표 후 집계 그래프 공개

### 2.3 익명 식별
- 최초 방문 시 서버가 익명 ID(UUID) 발급 → HttpOnly 쿠키 저장
- 닉네임은 익명 ID 기반 자동 생성 (예: "떠도는 두루미 3f2a") — 변경 불가
- 익명 ID별 활동 패턴은 어뷰징 탐지에만 사용

## 3. 페이지 구조

### 3.1 PC (해상도 1024px 이상)
- **3단 레이아웃**: 좌측 컬럼(파랑/좌 성향 피드) — 중앙 컬럼(플레이그라운드) — 우측 컬럼(빨강/우 성향 피드)
- 중앙 플레이그라운드: 태극 모티프의 혼합색(자주/보라 계열) 존
  - 상단: "오늘의 링" — 가장 뜨거운 매칭 주제 1건 크게 노출
  - 그 아래: 매칭된 글 쌍 카드 목록 (좌 반쪽 파랑 / 우 반쪽 빨강)
  - 카드 클릭 → 토론 페이지로 이동
- 토론 페이지: 상단에 좌/우 원글 요약 병렬 배치, 하단에 댓글 스레드 + 입장 투표 위젯

### 3.2 모바일 (1024px 미만)
- **하단 탭 3개**: [좌] [플레이그라운드(기본)] [우]
  - 플레이그라운드 탭이 항상 디폴트 진입점
  - 탭 색상: 좌=파랑, 우=빨강, 플레이그라운드=혼합색
- 플레이그라운드 탭: 글 쌍을 한 카드 안에 좌우 반반으로 표시
  (상하 배치 금지 — 위계 논란 방지. 카드 내부를 세로선으로 이등분)
- 좌/우 탭: 해당 성향 피드 리스트

### 3.3 공통 페이지
- 토론 상세 페이지 (댓글/투표)
- 이용약관 / 개인정보처리방침 / 서비스 소개(수집·출처 정책 공개)
- 404 / 점검 페이지

## 4. 데이터 수집 (크롤러)

### 4.1 전략
- **1단계**: RSS 공식 제공 소스부터 (예: 루리웹 뉴스 RSS). RSS 파서로 시작.
- **2단계**: robots.txt에서 허용된 경로만 대상으로 HTML 파싱 확장.
  - robots.txt 파싱 후 Disallow 경로는 절대 접근하지 않음 (코드로 강제)
  - User-Agent에 서비스명과 연락처 명시
  - 요청 간격: 사이트당 최소 10초 이상, 심야 배치 위주
- 소스별 성향 라벨(left/right)은 **관리자가 CMS에서 수동 지정** — 알고리즘 판단 금지
- 수집 대상 커뮤니티 목록은 코드가 아닌 DB(sources 테이블)에서 관리

### 4.2 크롤러 구성 (Python)
- `crawler/` 디렉토리, 독립 실행 스크립트
- 스택: `httpx` + `feedparser`(RSS) + `BeautifulSoup4`(HTML) + `psycopg`
- 스케줄: GitHub Actions cron 또는 Railway cron (10~30분 간격)
- 파이프라인: 수집 → 중복 체크(URL 해시) → 요약 추출(200자) → 저장 → 매칭 큐 등록
- 실패/차단 감지 시 해당 소스 자동 비활성화 + 관리자 알림

### 4.3 주제 매칭
- **처음부터 임베딩 기반** (토론 결정: TF-IDF는 짧은 커뮤니티 제목에 부정확):
  한국어 오픈소스 문장 임베딩 모델(ko-sbert 계열, `sentence-transformers`)을
  크롤러 배치에서 **로컬 실행** → API 비용 0원. 벡터는 pgvector에 저장.
- 코사인 유사도 임계값 이상 + 성향이 다른 글끼리 쌍 생성.
- 트래픽 성장으로 배치 시간이 문제되면 임베딩 API(OpenAI 등)로 전환 검토.
- 매칭 쌍은 관리자 CMS에서 수동 해제/수동 생성 가능.

## 5. 댓글·토론 시스템

- 구조: 글 쌍(topic_pair) 1 : N 댓글, 댓글 1 : N 대댓글(2 depth 제한)
- 작성 시: 익명 ID + 자동 닉네임 + 작성 시각 + (내부 저장용) IP 해시
- **셀프 입장 태그**: 댓글 작성 시 본인이 좌/우/중립 태그를 선택(선택사항) —
  태그 색상으로 댓글에 표시. **행동 패턴 기반 자동 성향 추정은 금지**
  (민감정보 프로파일링 리스크 + 낙인 효과. 토론 결정 사항)
- 정렬: 기본 "추천순", 옵션 "최신순"
- 금칙어/욕설 필터: 작성 시점에 1차 필터(사전 기반) → 걸리면 작성 차단 + 사유 안내
- 신고 누적 N건(기본 5) 도달 시 자동 블라인드 → CMS 검수 대기
- 입장 투표: 글 쌍당 익명 ID별 1회, 변경 가능, 실시간 비율 그래프 노출
- 레이트 리밋: 익명 ID당 댓글 분당 3개 / 시간당 20개, 투표·추천도 유사 제한

## 6. 여론조작·어뷰징 방지

- IP는 **원본 저장 금지** — salt + SHA-256 해시로만 저장, 90일 후 자동 파기
- 동일 IP 해시에서 다수 익명 ID 생성 감지 → 해당 그룹 추천/투표 가중치 하향
- 동일/유사 내용 반복 댓글(편집거리 기반) 자동 필터
- 시간대별 급증 패턴(예: 특정 글에 10분 내 추천 폭증) 이상 탐지 → CMS 알림
- 추천/비추천 집계 시 신뢰도 가중치 적용 (신규 익명 ID의 초기 가중치 낮음)
- reCAPTCHA v3 (점수 기반, UX 방해 없는 방식) — 댓글 작성 시에만

## 7. 법적·개인정보 요구사항 (필수 구현)

- 개인정보처리방침에 명시할 것:
  - 수집 항목: 쿠키(익명 ID), IP 해시, 접속 로그
  - 수집 목적: 어뷰징·여론조작 방지, 서비스 안정 운영
  - 보관 기간: IP 해시 90일, 로그 90일 후 파기
- 첫 방문 시 쿠키 사용 고지 배너 (동의 후 익명 ID 발급)
- 수집 글: 제목 + 200자 요약 + 출처 링크만. 이미지·본문 전체 미러링 금지.
- 각 커뮤니티의 삭제 요청 접수 채널(이메일) 운영 및 24시간 내 삭제 프로세스
- 광고 게재 시 애드센스 정책(성인/폭력 콘텐츠 옆 광고 금지) 준수 —
  블라인드된 토론 페이지에는 광고 미노출

## 8. 수익화 (광고)

- 플랫폼: 구글 애드센스 → 트래픽 성장 시 카카오 애드핏 병행 검토
- **단계적 오픈 (토론 결정)**: 정치 UGC는 애드센스 정책 리스크가 높으므로
  - 1단계: 정제된 요약만 노출되는 피드 페이지에만 광고 게재
  - 2단계: 모더레이션 지표(신고 처리율, 블라인드 비율) 안정화 확인 후 토론 페이지로 확대
  - 애드센스 심사 탈락/제한 대비 예비 채널로 카카오 애드핏 준비
- 광고 슬롯 (좌우 대칭 배치 필수):
  - PC: 좌 컬럼 피드 5번째 카드 뒤 1개 / 우 컬럼 동일 위치 1개 / 중앙 토론 페이지 댓글 상단 1개
  - 모바일: 각 탭 피드 5번째 카드 뒤 1개, 토론 페이지 댓글 상단 1개
- 광고 컴포넌트는 `<AdSlot position="..."/>`으로 추상화 — 위치별 노출/클릭 지표 수집
- 페이지 체류시간·스크롤 깊이 이벤트 수집 → 광고 위치 최적화에 활용

## 10. 인프라 / CI-CD (무료 티어 시작)

- **형상관리**: GitHub 모노레포
  ```
  /backend    (Spring Boot)
  /frontend   (React)
  /crawler    (Python)
  /infra      (Dockerfile, compose, 배포 설정)
  CLAUDE.md
  ```
- **CI**: GitHub Actions — PR마다 백엔드 테스트(Gradle) + 프론트 빌드 + 크롤러 lint
- **CD**:
  - 프론트: Vercel (main 브랜치 자동 배포)
  - 백엔드: Railway 또는 Fly.io (Dockerfile 배포)
  - DB: Supabase 무료 티어 (pgvector 기본 지원 — 임베딩 매칭에 필수. 토론 결정)
  - Redis: Railway Redis 또는 Upstash 무료 티어
  - 크롤러: GitHub Actions cron으로 시작 → 안정화 후 Railway cron 이전
- **모니터링**: Sentry 무료 티어(백/프론트), UptimeRobot(헬스체크)
- **성장 시 전환 계획**: AWS (ECS Fargate + RDS + ElastiCache + CloudFront) —
  본업이 AWS라서 전환 부담 낮음. 월 방문 10만 돌파 시점에 검토.

- 다크모드: 1차 릴리즈 제외 (백로그)

## 12. DB 스키마 (초안)

```sql
sources (id, name, side ENUM('left','right'), base_url, feed_url,
         crawl_type ENUM('rss','html'), enabled, created_at)

posts (id, source_id FK, title, summary VARCHAR(200), origin_url UNIQUE,
       origin_url_hash, published_at, collected_at, keywords TEXT[])

topic_pairs (id, left_post_id FK, right_post_id FK, similarity FLOAT,
             matched_by ENUM('auto','manual'), status ENUM('active','hidden'),
             created_at)

comments (id, pair_id FK NULL, post_id FK NULL, parent_id FK NULL,
          anon_id, nickname, body, ip_hash, status ENUM('visible','blinded','deleted'),
          up_count, down_count, created_at)

votes (id, pair_id FK, anon_id, stance ENUM('left','right','neutral'),
       updated_at, UNIQUE(pair_id, anon_id))

reactions (id, comment_id FK, anon_id, type ENUM('up','down'),
           UNIQUE(comment_id, anon_id))

reports (id, comment_id FK, anon_id, reason ENUM('abuse','hate','spam','etc'),
         detail, status ENUM('pending','accepted','rejected'), created_at)

anon_users (anon_id PK, first_seen, last_seen, ip_hash_recent,
            trust_score FLOAT DEFAULT 0.5, comment_count, vote_count)

```

### 19.2 보안 (Security)

- **현재 `npm audit` 상태**: postcss(moderate)/sharp(high) 취약점이 잡히지만
  둘 다 Next.js 내부 번들 이슈이고, `next/image`를 쓰지 않는 현재 코드에서는
  런타임에 노출되지 않음(D17). "next 9.3.3으로 다운그레이드"라는 audit의 자동
  제안은 명백히 무시할 것 — 그렇게 하면 오히려 훨씬 오래되고 취약한 버전이 됨.
- **아직 안 만든 것들 (M3에서 실제로 필요해짐)**:
  - CSRF: 쿠키 기반 익명 인증(`SameSite=Lax`, 9절)이라 어느 정도 방어되지만,
    댓글/투표/추천 같은 상태 변경 POST 엔드포인트는 Origin 헤더 검증 등
    추가 방어를 붙일 것.
  - Rate limiting의 실제 구현 지점: 5/6절에 정책(분당 3개/시간당 20개 등)은
    있지만 "어디서 강제하는지"(앱 레벨 인터셉터 vs Redis 카운터 vs
    게이트웨이)가 아직 미정. Redis는 9절에서 이미 스택으로 정해져 있으니
    앱 레벨에서 Redis 카운터로 구현하는 쪽으로 방향만 잡아둠.
  - XSS: React가 텍스트를 기본 이스케이프하지만, 댓글에 마크다운/리치텍스트를
    나중에 추가한다면(현재 계획엔 없음) `dangerouslySetInnerHTML` 류를 절대
    직접 쓰지 말 것 — 쓸 일이 생기면 그 시점에 별도로 검토.
  - 관리자 CMS 인증: 13절의 Basic Auth + IP 허용목록은 초기엔 충분하지만
    2FA가 없어 자격증명 유출에 취약 — 트래픽/운영자 늘어나면 재검토 대상.
- **Claude Haiku 신고 분류(D11/D14)**: 댓글 본문이 외부(Anthropic) API로
  전송되는 새로운 데이터 흐름이 생김 — 법적 부분(19.3)과 직결.
- CI에 의존성 취약점 자동 스캔(Dependabot 등)이 아직 없음 — D17 참고, 백로그.

### 19.3 법적 (Legal) — 웹서치 기반, 🔴 표시는 특히 변호사 확인 권장

- **실명확인제**: 정보통신망법 §44조의5(2014년 삭제), 공직선거법 §82조의6
  (2023년 삭제) 모두 헌재 위헌 결정 후 폐지되어 **완전 익명 댓글 시스템은
  현재 적법**. 3절/9절의 "익명 기반, 회원가입 없음" 설계는 법적으로 문제없음.
- 🔴 **정보통신망법 임시조치(§44조의2)**: "지체없이" 삭제/임시조치 의무는
  있으나 명시적 시간 기준(예: 24시간)은 법에 없음. D13에서 대응 프로세스를
  정했지만, "자동 블라인드"가 조문이 요구하는 개별 소명 확보·통지 요건을
  충분히 충족하는지는 판례로 명확히 확인되지 않음.
- **명예훼손 책임 범위**: 대법원 2008다53812 리딩케이스 기준, 플랫폼 운영자는
  사전 모니터링 의무 없이 **"구체적 인식(신고 등) + 방치"가 있을 때만 책임**
  (notice-and-takedown 원칙). 6/7절의 신고→검수 구조가 이 원칙에 부합하는
  방향이나, 신고 처리 이력을 증거로 남기는 게 중요(D13에 반영).
  참고: 2026.7.7. 시행 "허위조작정보 근절법"은 DAU 100만 이상 대상이라
  이 프로젝트 규모에선 해당 가능성 낮음(시행령 세부기준은 추후 재확인).
- 🔴 **개인정보보호법(PIPA)**: 익명ID/IP해시가 "개인정보"로 취급될지는
  단정하기 어려우나(가명정보 가능성 높음), 개인정보처리방침 게시 의무는
  **규모와 무관하게 항상 있음**(7절에 이미 계획됨, 유지). 1인 사업자도
  개인정보보호책임자가 대표자로 자동 지정됨(2024년 개정, 별도 지정 불요).
  Claude API로 신고 댓글을 보내는 것이 국외이전(§28조의8)에 해당하는지는
  댓글에 식별정보가 섞일 가능성 때문에 리스크가 있음 → D14로 처리방침에
  명시하기로 결정했으나, 정확한 법적 해당 여부는 확인 필요.
- **저작권/크롤링(4절)** — 5개 앵글로 재조사(2026-07-22, 최초 조사의 판례
  인용 오류를 정정함: 대법원 2021도1533은 "사람인 사건"이 아니라
  **야놀자 v. 여기어때 형사사건**이고, 사람인 관련 판례는 별개
  (서울고법 2017.4.6. 2016나2019365, 잡코리아 v. 사람인)):
  - **ToS 위반 ≠ 저작권/형사 위반, 별개 문제**. 대법원 2022.5.12. 2021도1533은
    이용약관이 원칙적으로 회원에게만 적용되고 비회원(크롤러)에겐 계약이
    성립하지 않아 구속력을 묻기 어렵다고 봄. 정보통신망법 §48(침입죄)의
    "접근권한" 유무도 주관적 의사가 아니라 **기술적 보호조치 + 이용약관 등
    객관적 사정**으로 판단 — robots.txt 준수 + 로그인 없는 공개 페이지만
    수집하면 침입죄 불성립 가능성이 높음(위 판결 무죄 확정 근거).
  - 단, 같은 사안의 **민사에서는 여기어때가 부정경쟁방지법 카목(성과물
    무단사용)으로 10억원 배상 인정**(1·2심) — 형사 무죄와 민사 책임이
    갈린 사례. 카목은 "경쟁관계" 요건이 있어, 소스가 늘어나 원 커뮤니티와
    직접 경쟁 구도로 보일 수 있는 M2 시점에 재검토 필요.
  - **잡코리아 v. 사람인**(서울고법 2016나2019365)은 대량·구조적 DB 통째
    수집이 저작권법상 DB제작자 권리 침해로 인정된 별개 사례 — "양적·질적
    상당 부분 복제"가 기준이라, 제목+200자 요약+링크만 저장하는 이 프로젝트
    구조는 이 유형의 리스크에서는 상당히 벗어남.
  - **제목/짧은 문구는 통상 저작물성 부정**, 저작권법 §7⑤(사실 전달 시사보도)
    예외도 있으나 이는 언론 보도 조항이라 개인 의견이 섞인 커뮤니티 게시글에
    그대로 적용될지는 불확실. **핵심 안전장치는 "발췌"가 아니라 "재작성"** —
    200자 요약이 원문 표현을 그대로 가져오면(직접 발췌) 복제/2차적저작물
    작성권 침해 소지가 커지고, 사실관계만 자체 문장으로 재구성하면 리스크가
    낮음 — 크롤러 구현 시(`summarize.py`) 원문 문장을 그대로 잘라내는 방식이
    아니라 재서술 방식으로 가는 게 안전.
  - 딥링크(원문 링크) 자체는 복제·전송이 아니라는 게 원칙이며, 대법원
    2021.9.9. 2017도19025 전합의 "링크 방조" 법리는 **불법 게시물**에
    링크하는 경우에 한정 — 합법적으로 공개된 원 커뮤니티 글에 링크하는 이
    프로젝트엔 해당 안 됨.
  - "제목+요약+링크만으로 구성된 큐레이션 서비스"가 이 형태 그대로 분쟁화된
    국내 판례는 확인 안 됨(5개 조사 공통 결론) — 실무 리스크는 낮은 편이나
    확정된 판례로 보장되는 영역은 아님. 🔴 소스를 여러 개로 늘리는 M2 진입
    전, 특히 "robots.txt에 크롤링 명시적 금지" 문구가 있는 커뮤니티는
    제외하는 걸 원칙으로 하고, 지재권 전문 변호사 확인을 한 번 받는 걸 권장.
- **선거법**: §82조의6(실명확인) 삭제로 그 리스크는 사라졌지만, §82조의4⑤에
  따라 선관위/후보자가 위법 게시물에 대해 "홈페이지 관리·운영자"에게도 직접
  삭제를 요청할 수 있음 — 16절 D10의 "선거 모드" CMS 토글과 별개로, 선거
  임박 시점엔 선관위에 사전 문의하는 절차를 M4 CMS 작업 시 문서화할 것.

  들어와도 D13의 "영업일 기준 24시간" SLA를 지키려면 최소한의 알림 체계

  M4 CMS 작업 시 admin_alerts 테이블/알림 채널과 함께 설계)가 있어야 함.

  자동으로 되는지 아직 확인 안 됨 — 유료 전환(D15/19.1) 시점에 백업 정책을
  같이 확정할 것.
- **모더레이션 용량**: D11(LLM 1차 분류)로 확장성은 확보했지만 최종 판단은
  여전히 1인 — "신고 처리 지연 N건 이상 누적 시 신규 댓글 임시 중단" 같은
  안전판 규칙이 아직 없음. M3 설계 시 추가 검토.
- **의존성/보안 패치 프로세스**: 19.2의 Dependabot 미도입과 동일 이슈 —
  1인 운영이라 수동으로 챙기기 어려우니 자동화가 특히 중요.
- **문서화 부재**: 공동 운영자가 생기거나 사고 발생 시 인수인계할 런북이
  없음 — 지금은 1인 사이드 프로젝트라 우선순위 낮음, 트래픽/운영자가 늘면
  재검토.
